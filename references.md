# PostgreSQL Keys & References

A quick reference for **Primary Keys**, **Foreign Keys**, and **ON DELETE** actions in PostgreSQL, using a `users → photos` example.

---

## Schema

```sql
CREATE TABLE users (
  id       SERIAL PRIMARY KEY,
  username TEXT NOT NULL
);

CREATE TABLE photos (
  id     SERIAL PRIMARY KEY,
  url    TEXT NOT NULL,
  userId INTEGER DEFAULT NULL REFERENCES users(id) ON DELETE SET DEFAULT
);
```

---

## Primary Key (PK)

A primary key uniquely identifies every row in a table. It combines two constraints in one declaration: `UNIQUE` and `NOT NULL`.

```sql
id SERIAL PRIMARY KEY
```

| Property | Detail |
|---|---|
| Uniqueness | No two rows can share the same PK value |
| Not null | A PK column can never be `NULL` |
| Auto-index | PostgreSQL automatically creates a unique index on the PK column |
| Per table | Each table can have at most **one** primary key |
| `SERIAL` | Shorthand for an auto-incrementing integer (`GENERATED BY DEFAULT AS IDENTITY` is the modern equivalent) |

### PK variants

```sql
-- Single column (most common)
id SERIAL PRIMARY KEY

-- Modern equivalent (PG 10+)
id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY

-- Composite PK — declared as a table constraint
PRIMARY KEY (order_id, product_id)

-- Named constraint
CONSTRAINT pk_users PRIMARY KEY (id)
```

### Adding / dropping a PK

```sql
-- Add after creation
ALTER TABLE users ADD PRIMARY KEY (id);

-- Drop (rarely needed)
ALTER TABLE users DROP CONSTRAINT users_pkey;
```

## Foreign Key (FK)

A foreign key links a column in one table to the **primary key** (or any unique column) of another table. PostgreSQL enforces that every non-null FK value must exist in the referenced table.

```sql
userId INTEGER DEFAULT NULL REFERENCES users(id) ON DELETE SET DEFAULT
--     ^^^^^^^              ^^^^^^^^^^^^^^^^^^^^^ ^^^^^^^^^^^^^^^^^^^^^
--     column type          references target      delete behaviour
```

| Property | Detail |
|---|---|
| Referential integrity | Cannot insert a `userId` that doesn't exist in `users.id` |
| Nullable | A FK column can be `NULL` (meaning "no parent") unless `NOT NULL` is added |
| No auto-index | Unlike PKs, FK columns are **not** automatically indexed — add one manually |
| Multiple FKs | A table can have as many FK columns as needed |

### FK variants

```sql
-- Inline (shorthand)
userId INTEGER REFERENCES users(id)

-- Inline with default and delete action
userId INTEGER DEFAULT NULL REFERENCES users(id) ON DELETE SET DEFAULT

-- Named table-level constraint (preferred for complex schemas)
CONSTRAINT fk_photos_user
  FOREIGN KEY (userId)
  REFERENCES users(id)
  ON DELETE SET DEFAULT

-- Composite FK
CONSTRAINT fk_order_item
  FOREIGN KEY (order_id, product_id)
  REFERENCES order_items(order_id, product_id)
```

### Adding / dropping a FK

```sql
-- Add after creation
ALTER TABLE photos
  ADD CONSTRAINT fk_photos_user
  FOREIGN KEY (userId) REFERENCES users(id)
  ON DELETE SET DEFAULT;

-- Drop
ALTER TABLE photos DROP CONSTRAINT fk_photos_user;
```

### Index the FK column manually

```sql
-- PostgreSQL does NOT auto-index FK columns
CREATE INDEX idx_photos_userId ON photos(userId);
```

Without this index, every JOIN on `photos.userId` triggers a full table scan.

## ON DELETE Actions

Defines what happens to child rows in `photos` when the referenced parent row in `users` is deleted.

| Action | What happens to `photos.userId` | Blocks delete? |
|---|---|---|
| `NO ACTION` *(default)* | Nothing — error is raised | Yes |
| `RESTRICT` | Same as `NO ACTION`, checked immediately (not deferrable) | Yes |
| `CASCADE` | Child row is **deleted** automatically | No |
| `SET NULL` | FK column is set to `NULL` | No |
| `SET DEFAULT` | FK column is set to its **column default** | No |

> This schema uses **`SET DEFAULT`** with `DEFAULT NULL`, so deleting a user sets the orphaned photo's `userId` to `NULL`.

### Side-by-side example

Given: `users` has `id=1` (thor). `photos` has a row with `userId=1`.

```sql
DELETE FROM users WHERE id = 1;
```

| Action on FK | Result in photos |
|---|---|
| `NO ACTION` / `RESTRICT` | ❌ Error — delete is blocked |
| `CASCADE` | 🗑️ The photo row is also deleted |
| `SET NULL` | `userId` → `NULL` |
| `SET DEFAULT` | `userId` → column default (here: `NULL`) |

### ON UPDATE actions

The same keywords apply to `ON UPDATE` (triggered when the parent PK value changes):

```sql
userId INTEGER REFERENCES users(id)
  ON DELETE SET DEFAULT
  ON UPDATE CASCADE   -- if users.id changes, mirror it into photos.userId
```

## Walkthrough

### 1. Seed data

```sql
INSERT INTO users (username) VALUES ('thor'), ('loki'), ('iron_man');
-- id: 1=thor, 2=loki, 3=iron_man

INSERT INTO photos (url, userId) VALUES
  ('https://.../bird.jpg',         1),  -- owned by thor
  ('https://.../strawberry.jpg',   3);  -- owned by iron_man
```

### 2. Delete a referenced user

```sql
DELETE FROM users WHERE id = 1;  -- deletes thor
```

`ON DELETE SET DEFAULT` fires → `photos.userId` for the bird photo is set to `NULL`.

### 3. Result

```sql
SELECT * FROM users;
--  id | username
-- ----+----------
--   2 | loki
--   3 | iron_man

SELECT * FROM photos;
--  id | url                | userid
-- ----+--------------------+--------
--   1 | .../bird.jpg       | NULL    ← was 1 (thor), reset to DEFAULT (NULL)
--   2 | .../strawberry.jpg | 3
```

## Common Gotchas

- **`SET DEFAULT` needs a safe default** — if the default is a non-null FK value that doesn't exist, the delete will still fail. `DEFAULT NULL` is the safest choice.
- **No auto-index on FK columns** — always add `CREATE INDEX` manually on FK columns used in JOINs.
- **`RESTRICT` vs `NO ACTION`** — both block the delete, but `RESTRICT` is checked immediately; `NO ACTION` can be deferred with `DEFERRABLE INITIALLY DEFERRED` for circular FK situations.
- **Composite PKs and FKs** — both sides must match in column count and types exactly.
- **`SERIAL` is legacy** — prefer `GENERATED ALWAYS AS IDENTITY` in new schemas (PG 10+); it's SQL-standard and safer against manual inserts overriding the sequence.
